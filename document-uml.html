Introducción al Modelo de Datos de Nexus

El presente documento detalla la estructura fundamental de la información que la aplicación Nexus gestionará, tal como se representa en el Diagrama de Clases UML a continuación. Este diagrama funciona como el "plano" conceptual de la base de datos y la lógica de negocio, mostrando las entidades clave de la aplicación, los datos que cada una almacena (sus atributos) y cómo se interconectan entre sí (sus relaciones).

Entender este modelo es vital para cualquier etapa del desarrollo. Nexus, al combinar ofertas de productos nuevos con un mercado de segunda mano, maneja una amplia variedad de información, incluyendo perfiles de usuario, anuncios de productos, interacciones sociales y sistemas de comunicación.

Esta versión del modelo es la más completa y precisa hasta la fecha, reflejando fielmente el diseño del diagrama UML proporcionado. Incorpora identificadores enteros para simplificar, una denominación clara para las ofertas de usuario (OfertaUsuario), una gestión estructurada de los tipos de ofertas administradas, la inclusión de una funcionalidad de ListaFavoritos para la organización personalizada de contenido, y la tipificación de estados de producto mediante un enumerador (ESTADOPRODUCTO).

Clases Principales y sus Atributos

Las clases son los componentes básicos de nuestro sistema, representando los tipos de objetos o conceptos fundamentales que la aplicación necesita gestionar y almacenar.

1. Gestión de Usuarios y Perfiles

USUARIO (Clase Base)

Propósito: Representa a cualquier persona o entidad con una cuenta registrada en la plataforma Nexus.

  • id: Integer: Identificador único.
  • user: String {único}: Alias público único.
  • email: String {único}: Email para login y comunicaciones.
  • password: String: Contraseña cifrada.
  • fechaRegistro: DateTime: Fecha de registro.
  • rol: Enum<"Particular", "Empresa_Publicidad", "Admin">: Tipo de acceso.

Funcionalidades: registrarse(datos), iniciarSesion(credenciales), editarPerfil(datos).

PERFIL

Propósito: Información pública y personalizable del usuario.

  • id: Integer: Identificador único.
  • esVerificado: Boolean: Indicador de verificación.
  • fotoPerfil: String (URL): Imagen de perfil.
  • biografia: String (opcional): Texto descriptivo.
  • ubicacion: String: Ubicación geográfica.
  • reputacion: Integer (0-5 estrellas): Valoración de la comunidad.
PARTICULAR (Subclase de USUARIO)

Propósito: Usuario individual para compra/venta y subida de ofertas.

  • telefono: String (opcional): Contacto opcional.
EMPRESAPUBLICIDAD (Subclase de USUARIO)

Propósito: Empresa colaboradora para publicidad o afiliación.

  • nombreComercial: String
  • cif: String {único}
  • contactoPrincipal: String
  • emailContacto: String
  • tipoServicioContratado: Enum<"Publicidad_Banner", "Comision_Afiliacion">
  • fechaContrato: DateTime

2. Gestión de Contenido (Productos y Ofertas)

CONTENIDOPUBLICABLE (Clase Abstracta)

Propósito: Base común para anuncios y ofertas con interacciones.

  • id: Integer
  • titulo: String
  • fechaPublicacion: DateTime
  • numLikes: Integer
  • numComentarios: Integer

Funcionalidades: publicar(), eliminar().

ANUNCIO (Subclase de CONTENIDOPUBLICABLE)

Propósito: Artículo de segunda mano en venta.

  • descripcion: String
  • estadoProducto: ENUM_ESTADOPRODUCTO
  • precio: Double
  • fechaModificacion: DateTime
  • ubicacion: String
  • estaVendido: Boolean
OFERTAUSUARIO (Subclase de CONTENIDOPUBLICABLE)

Propósito: Oferta/descuento de producto nuevo compartida por usuarios.

  • urlOferta: String
  • tienda: String
  • precioOriginal: Double (opcional)
  • precioOferta: Double
  • fechaExpira: DateTime (opcional)
OFERTA_ADMINISTRADOR

Propósito: Contenido promocional gestionado por Nexus.

  • id: Integer
  • titulo: String
  • descripcion: String (HTML)
  • urlDestino: String
  • fechaInicio: DateTime
  • fechaFin: DateTime
  • estaActiva: Boolean
  • tipoOferta: TIPO_OFERTA_ADMIN
(ENUM) TIPO_OFERTA_ADMIN

Propósito: Tipos de presentación para OFERTA_ADMINISTRADOR.

  • Valores: Banner, Destacado_Feed, Patrocinado.
(ENUM) ESTADOPRODUCTO

Propósito: Condición de un producto en ANUNCIO.

  • Valores: ComoNuevo, Usado, Defectos.

3. Clases Auxiliares para Contenido y Organización

IMAGEN

Propósito: Fotografía asociada a un ANUNCIO.

  • id: Integer
  • url: String (URL)
CATEGORIA

Propósito: Clasifica ANUNCIO y OFERTAUSUARIO.

  • id: Integer
  • nombre: String {único}
  • icono: String (URL)

4. Clases de Interacción Social

COMENTARIO

Propósito: Interacción textual sobre CONTENIDOPUBLICABLE.

  • id: Integer
  • texto: String
  • fecha: DateTime
  • esReportado: Boolean

Funcionalidades: reportar().

LIKE

Propósito: Acción "me gusta" sobre CONTENIDOPUBLICABLE.

  • id: Integer
  • fecha: DateTime
LISTAFAVORITOS

Propósito: Permite guardar CONTENIDOPUBLICABLE.

  • id: Integer
  • nombre: String
  • fechaCreacion: DateTime
  • esPublica: Boolean

Funcionalidades: añadirContenido(), eliminarContenido(), exportar().

5. Clases de Comunicación

CHAT

Propósito: Conversación privada entre dos PERFILES.

  • id: Integer
  • fechaCreacion: DateTime
  • estaActivo: Boolean
MENSAJE

Propósito: Mensaje individual dentro de un CHAT.

  • id: Integer
  • texto: String
  • fechaEnvio: DateTime
  • leido: Boolean

Relaciones entre Clases (Asociaciones)

Las relaciones definen cómo las clases se conectan. La multiplicidad ("1", "*", "2") indica cuántas instancias se relacionan.

1. Relaciones de Usuario y Perfil:

  • USUARIO posee PERFIL (1 a 1)
  • PARTICULAR es un USUARIO (Herencia)
  • EMPRESAPUBLICIDAD es un USUARIO (Herencia)
  • PERFIL crea ANUNCIO (1 a *)
  • PERFIL sube OFERTAUSUARIO (1 a *)
  • PERFIL da LIKE (1 a *)
  • PERFIL escribe COMENTARIO (1 a *)
  • PERFIL participa en CHAT (* a 2)
  • PERFIL crea LISTAFAVORITOS (1 a *)

2. Relaciones de Contenido y sus Auxiliares:

  • ANUNCIO es un CONTENIDOPUBLICABLE (Herencia)
  • OFERTAUSUARIO es un CONTENIDOPUBLICABLE (Herencia)
  • ANUNCIO tiene IMAGEN (1 a *)
  • ANUNCIO pertenece a CATEGORIA (1 a 1)
  • OFERTAUSUARIO pertenece a CATEGORIA (1 a 1)
  • OFERTA_ADMINISTRADOR tiene TIPO_OFERTA_ADMIN (1 a 1)

3. Relaciones de Interacción con Contenido:

  • COMENTARIO sobre CONTENIDOPUBLICABLE (1 a 1)
  • LIKE a CONTENIDOPUBLICABLE (1 a 1)
  • LISTAFAVORITOS contiene CONTENIDOPUBLICABLE (* a *)

4. Relaciones de Comunicación:

  • CHAT contiene MENSAJE (1 a *)
  • MENSAJE enviado por PERFIL (* a 1)
